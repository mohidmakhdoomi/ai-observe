### 2. Technical Feasibility

**Verified against codebase — all feasible:**

- **`TraceParser` reuse**: The spec proposes feeding lines incrementally into a shared `TraceParser`. Looking at `parse_lines()` (line 112), it iterates an `Iterable[str]` and appends to `self.events`. A `feed_lines` method that does the same but returns newly-added events is trivially implementable. ✅
- **`safe_write_jsonl` reuse for fallback**: The existing function (line 212) uses `O_WRONLY | O_CREAT | O_TRUNC | O_NOFOLLOW` — perfect for the overwrite-on-fallback case. ✅
- **Thread safety**: The spec correctly notes the main thread is blocked in `proc.wait`, so there's no contention with the parser thread over shared state. The only shared state is the stop flag (set by main, read by thread) — a simple `threading.Event`. ✅
- **`ParserFailure` propagation from thread**: The spec says "catch `ParserFailure` in the thread, re-raise it after join." This is clean — store the exception in a thread-accessible attribute, check it after `join()`. ✅
- **Python stdlib only**: Everything described (threading, file I/O, os module) is stdlib. ✅

**One technical nuance worth noting:**

The spec says the live JSONL writer opens the pre-created `.jsonl` with `O_WRONLY | O_APPEND | O_NOFOLLOW`. But on `ParserFailure`, the spec says `.jsonl` should be "truncated to zero bytes" via `safe_write_jsonl` with an empty event list. Since `safe_write_jsonl` uses `O_TRUNC`, it will correctly zero the file. However, the live writer thread may still have the file open with `O_APPEND` — the thread needs to close its handle *before* the fallback truncation. The spec implies this (thread completes/errors → main handles fallback) but doesn't explicitly state the thread must close its `.jsonl` handle in all exit paths (clean exit, exception, `ParserFailure`). This should be clarified, though a builder would likely get it right since the thread's cleanup naturally closes the handle.

### 3. Edge Cases and Error Handling

**One edge case not explicitly addressed:**
- **Rapid successive events written by strace in a single `read()` call**: The spec says `read(64KB)` and split on newline. If strace flushes a large batch (say 100 lines at once), the parser thread processes them all in one loop iteration and writes 100 JSONL lines. This is fine behavior-wise but worth confirming the spec intends no event batching/buffering on the write side — every event gets its own `write()+flush()`. The spec does say this ("writes one line per event with `fh.write(...)` followed by `fh.flush()`") so this is covered. ✅

### 6. Ambiguities

**Minor:**
- The spec says `CODEV_OBSERVE_LIVE_POLL_MS` has "Lower bound `10`, upper bound `2000`." It also says "Out-of-range or unparseable values fall back to the default." But the test scenario says testing with `0` and `9999` — `0` is below the lower bound and `9999` is above the upper bound, so this is consistent. However, the spec doesn't say whether `10` and `2000` themselves are valid (inclusive bounds) — a builder would assume inclusive, which is fine.
- The spec says the live writer uses `O_APPEND` but the post-hoc `safe_write_jsonl` uses `O_TRUNC`. The spec should clarify that the live writer uses a *different* open helper (not `safe_write_jsonl` itself) that shares the same safety checks but uses `O_APPEND` instead of `O_TRUNC`. It does say "through a new helper that runs the same `verify_log_path_safe`/symlink rejection" — this is clear enough.
