KEY_ISSUES:
- **Startup failure behavior is internally inconsistent.** In phase 2, `LiveTracer.start()` failure is described as “print warning, discard tracer, and fall back to the Spec 1 post-hoc-only path,” which implies normal exit-code behavior. But the planned test `test_live_start_open_failure_falls_back_to_post_hoc` expects strict mode to exit `1` and stderr to mention the original Codex exit first. The plan needs to explicitly choose and document one contract here, then show how `run()` tracks that startup failure through final exit-code handling.
- **`LiveTracer.start()` needs explicit cleanup on partial startup failure.** In the current codebase, `prepare_logs()` pre-creates both files and `safe_write_jsonl()` is careful about fd ownership. Your planned `start()` opens `.trace`, then `.jsonl`, then starts the thread, but says it “does not catch — it propagates.” If the second open or `thread.start()` fails, the first handle leaks. Add a startup `try/except`/`finally` cleanup path so partially opened fds are closed before propagating.
- **The `CODEV_OBSERVE_LIVE_PARSE=0` regression coverage is incomplete versus the spec.** The spec says disabled live mode should produce the same `.jsonl` **and stderr output** as today. The plan’s test only checks that no live thread starts and the final JSONL matches post-hoc parsing. Add an explicit stderr-equivalence assertion, or state that the disabled path is literally the existing branch with no new warnings/messages.
- **One phase-1 test expectation is slightly off for the actual code shape.** `safe_append_jsonl_handle()` reopens an already-created `.jsonl`; it does not create the file. So “opens with mode `0o600`” is not really something that helper can guarantee by itself the way `exclusive_touch()`/`prepare_logs()` do. Reword that success criterion/test to check “does not widen permissions” or rely on the pre-created `0600` artifact instead.
