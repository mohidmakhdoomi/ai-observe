### Issues and Suggestions

**1. Join-timeout + daemon thread: potential for truncated JSONL line (Minor)**

When the join times out (lifecycle step 7), the daemon thread is still alive and potentially mid-write to `.jsonl`. When the process exits, the daemon thread is killed. Since writes are line-at-a-time with `flush()`, the window is tiny — but a consumer could see a partial final JSON line. The spec's note about "leaving partial .jsonl" acknowledges this, but a builder might want explicit guidance: should the consumer tolerate a truncated last line? Consider noting this as a documented limitation for `.jsonl` consumers.

**2. `CODEV_OBSERVE_LIVE_PARSE` value semantics (Trivial)**

The spec says `CODEV_OBSERVE_LIVE_PARSE=0` disables live mode. It doesn't explicitly state what values enable it — presumably anything other than `"0"` (including absence). This matches the project's pattern for `CODEV_OBSERVE_DISABLE`, `CODEV_OBSERVE_QUIET`, etc., so a builder should infer it, but an explicit note wouldn't hurt.

**3. The `feed_lines` API shape is underspecified (Intentional)**

The spec says "Add a generator-style entry point or a `feed_lines(iterable)`-like helper" — this leaves the exact API to the plan phase, which is appropriate for a spec. The key invariant (same `TraceParser` instance, same ordering, events accessible after each call) is clearly stated.

**4. No mention of Spec 2 (Nice-to-know)**

The spec numbering jumps from 1 to 3. If Spec 2 exists and touches `codex_observe.py` or `trace_parser.py`, there could be merge conflicts. Not a spec issue per se, but worth confirming no concurrent spec modifies the same files.
