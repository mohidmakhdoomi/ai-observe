### 4. Potential Issues

**Issue A — Phase 2 `_make_event` compatibility concern**: When Phase 2 bumps `SCHEMA_VERSION` to 2 and adds `source`/`confidence` to `_make_event()`, the `tests/_aggregator_oracle.py` (Python mirror of `aggregator.js`) may break if it checks for `schema_version: 1`. The plan mentions Phase 5 updates `tests/_aggregator_oracle.py` but Phase 2's schema change will land first. A builder needs to know: either (a) the aggregator oracle must tolerate v2 events from Phase 2 onward, or (b) Phase 2 must update the oracle minimally for passthrough.

Looking at this more carefully, the aggregator oracle tests in Phase 5 are explicitly listed as updating `tests/_aggregator_oracle.py` "in lock-step with `aggregator.js`" — but the aggregator only sees *sanitized* events from the tailer, which currently strip `schema_version`. So this is actually fine as long as Phase 2 updates the sanitizer to pass through `source`/`confidence` but NOT `schema_version` (which the spec says should be included in sanitized output). The plan says "sanitize `schema_version`, `source`, and `confidence` to the browser" — this will flow through the aggregator. **A builder would benefit from an explicit note that Phase 2 must verify aggregator tests still pass after adding these new fields to sanitized output, or that Phase 2 should update the oracle to ignore unknown fields.**

This is a minor gap, not a blocker — a competent builder will run the test suite and fix forward.

**Issue B — `<session>.meta.json` write safety**: Phase 1 introduces `<session>.meta.json` writes. The existing `safe_write_jsonl()` and `verify_log_path_safe()` enforce symlink safety and path containment. The plan says to write meta.json but doesn't explicitly say to use the same safe-write infrastructure. This is implicit but worth calling out — a builder should use the same safety primitives.

**Issue C — Phase 3/4 boundary for dedup helpers**: The plan says Phase 3 SHOULD include dedup/correlation helpers, and Phase 4 says "add wrapper-facing helpers for deduplication/correlation... if not completed in Phase 3." This is deliberately flexible, which is good for builder autonomy, but the builder must ensure Phase 3 is independently testable without dedup — which the plan's Phase 3 tests confirm (they only test manifest diff, not dedup). Clean.
