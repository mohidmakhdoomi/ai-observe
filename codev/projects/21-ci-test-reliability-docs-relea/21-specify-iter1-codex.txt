KEY_ISSUES:
- The main ambiguity is CI artifact validation: `tests/test_packaging_smoke.py` currently builds and installs its **own** artifacts in `setUpModule()`, so the spec should clarify whether CI must also explicitly test the workflow-built wheel/sdist, or whether satisfying the smoke harness’s gates is sufficient. As written, an implementation could “build artifacts” and separately run the smoke harness without ever proving the workflow-built artifacts were the ones exercised.
- Related minor ambiguity: if CI runs `python -m unittest discover -s tests`, that already includes `test_packaging_smoke.py`. The spec should state whether packaging smoke is expected to run as part of the main suite, as a distinct step, or both, to avoid accidental double-running or a capability-gated skip being mistaken for success.
