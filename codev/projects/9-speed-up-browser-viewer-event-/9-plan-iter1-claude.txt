### 3. Technical Approach

**Phase 1 — Linear tailer:** The plan correctly identifies the quadratic behavior in `_poll_once()`:
```python
self._buf += chunk
while True:
    nl = self._buf.find(b"\n")
    ...
    line, self._buf = self._buf[:nl], self._buf[nl + 1:]
```
Each iteration slices the remainder of `self._buf`, creating a new bytes object. For an 80k-line / 409MB chunk, this is O(n²) in total bytes copied. The plan's approach — splitting complete lines in one pass and retaining only the final incomplete fragment — is the right fix. A straightforward implementation would use `bytes.split(b'\n')` or `memoryview` to avoid repeated copies.

**Phase 2 — SSE batching:** The plan correctly targets the existing `_send_batch` → `_send_event` loop that flushes per event. Introducing `append_batch` as a new SSE event type carrying a JSON array with one flush per batch is sound. The plan wisely notes:
- Empty batches should not be emitted
- Sparse live events should not be delayed to fill a batch
- Batch size should be bounded to avoid blocking browser `JSON.parse`

The existing server architecture (snapshot watermark + slice + loop) maps cleanly to batched delivery — `broadcaster.slice(n, new_end)` already returns arrays.

**Phase 3 — Browser:** The plan correctly identifies that `pruneSelections` calls `collectTreePaths` which recursively walks the entire snapshot tree on every render, even when `state.selectedPaths.size === 0`. The fast-path check is trivially correct and safe. The `append_batch` listener is straightforward — parse JSON array, loop, push to `eventBuffer`, ingest into aggregator. The existing `scheduleRender()` 250ms debounce handles coalescing.

**No obvious better approaches are missed.** The plan aligns with Approach B from the spec, which is well-justified by measurements.
