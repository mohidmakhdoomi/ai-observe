This PR implements performance improvements for the `ai_observe.viewer` targeting large JSONL backlogs (~80k events / 409MB). It was executed through a well-structured 4-phase ASPIR protocol: linear tailer, SSE+browser batching, selection pruning fast path, and documentation/regression.

### 1. Code Quality and Correctness ✅

**Tailer (tailer.py):** The core fix is clean and correct. The old quadratic `while True: line, self._buf = self._buf[:nl], self._buf[nl+1:]` loop is replaced by a linear index-cursor scan:
```python
data = self._buf + chunk if self._buf else chunk
start = 0
while True:
    nl = data.find(b"\n", start)
    if nl < 0: break
    self._handle_line(data[start:nl])
    start = nl + 1
self._buf = data[start:] if start < len(data) else b""
```
The `if self._buf else chunk` optimization avoids a gratuitous concat when the buffer is empty — the common case on startup. Total work is O(bytes + lines). Partial trailing line behavior, error handling, and all rotation/truncation paths are unchanged.

**Server (server.py):** `_event_batches()` is a clean module-scope generator with input validation (`batch_size > 0`). `_send_append_batches()` routes through it into `append_batch` SSE frames. The broadcaster watermark model is unchanged — batching is purely a delivery optimization.

**Browser (index.js):** The `ing1`/`ingN`/`ingestAppendData`/`ingestAppendBatchData` helpers are well-factored, testable, and exported for Node. `ingN` guards with `Array.isArray`. Both frame listeners share `onIngested(n)` which only updates timestamp/schedules render when events were actually ingested.
